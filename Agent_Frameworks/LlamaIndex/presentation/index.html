What is LlamaIndex?

🗂️ RAG-First Architecture

Built around Retrieval-Augmented Generation as a first-class primitive. VectorStoreIndex, SummaryIndex, SubQuestionQueryEngine, and RouterQueryEngine are built-in — not bolted on from a separate library.

⚡ Event-Driven Workflow

The Workflow class routes execution via Python type annotations on @step parameters. No explicit routing tables, no string-keyed edges — the type system IS the router.

🧩 Modular Ecosystem

Over 160 integrations as independent pip packages (llama-index-llms-openai, llama-index-agent-openai, etc.). Install only what you need — no monolithic dependency.

🔍 Native Observability

LlamaDebugHandler intercepts every LLM call, embedding, and retrieval event. Provides token counts and latency traces with zero extra code — no external tracing library needed.

🤖 Multiple Agent Patterns

OpenAIAgent (function-calling), ReActAgent (text-based ReAct), and AgentWorkflow (orchestration). Each works with any QueryEngineTool or FunctionTool.

💬 Memory Management

ChatMemoryBuffer provides token-limited conversation history. Works across async steps via the workflow Context store — no external vector DB needed for short sessions.

Type-Based Dispatch

Steps declare what event they consume via their parameter type annotation. The Workflow engine dispatches automatically — no routing table, no string keys.

@step async def rag_analysis_step( self, ctx: Context, ev: WebResearchDoneEvent # ← type IS the route ) -> RagAnalysisDoneEvent: ...

ProgressEvent Streaming

ctx.write_event_to_stream(ProgressEvent(...)) surfaces live updates in stream mode — similar to LangGraph's StreamWriter but typed.

Context Store

await ctx.set("key", value) / await ctx.get("key") — a shared async key-value store accessible to all steps in the workflow.

The RAG Pipeline

📄 VectorStoreIndex

Builds an in-memory vector index from web-gathered text using text-embedding-3-small. Uses LlamaIndex's SimpleVectorStore — no C++ FAISS build required. Swap to FAISS, Pinecone, or Chroma with one import change.

🧩 QueryEngineTool

Wraps any query engine as a tool that agents or composite engines can call. The bridge between indexes and agents — unique to LlamaIndex.

QueryEngineTool.from_defaults( query_engine=index.as_query_engine(), name="research_knowledge_base", description="Contains research about...", )

🔢 SubQuestionQueryEngine

LlamaIndex's signature feature. Given a complex query, it automatically decomposes it into sub-questions, queries the vector index for each, then synthesizes a unified answer. No equivalent in LangGraph or CrewAI.

🔀 RouterQueryEngine

Routes each incoming query to the best of multiple query engines using LLMSingleSelector. The LLM reads each engine's description and selects the best match. In our system: vector (precise retrieval) vs. summary (broad synthesis).

RouterQueryEngine( selector=LLMSingleSelector.from_defaults(), query_engine_tools=[vector_tool, summary_tool], )

📋 SummaryIndex

A sequential index optimized for summarization — traverses all nodes to produce a full-document answer. Paired with VectorStoreIndex in the RouterQueryEngine for complementary strengths.

🏗️ Structured Output

llm.as_structured_llm(ResearchReport) binds a Pydantic model to an LLM — every response is automatically parsed and validated. Same concept as LangChain's with_structured_output.

Agent Roles

🧠 Orchestrator
gpt-4o-mini · temperature=0 · no tools

Decomposes the research topic into 3–5 focused sub-questions. Uses llm.acomplete() directly — pure LLM reasoning, no tools needed. Sets the agenda for all downstream steps.

🌐 Web Researcher
gpt-4o-mini · FunctionAgent

Runs one FunctionCallingAgent call per sub-question. Collects raw text chunks for indexing. Demonstrates FunctionTool.from_defaults wrapping plain Python functions.

duckduckgo_search wikipedia_search scrape_webpage
📚 RAG Analyst
gpt-4o · text-embedding-3-small · SubQuestionQueryEngine

Builds VectorStoreIndex from web chunks, wraps it as QueryEngineTool, and runs SubQuestionQueryEngine to answer each planned sub-question with evidence-backed responses.

🔀 Synthesizer
gpt-4o · RouterQueryEngine · LLMSingleSelector

Creates both a VectorStoreIndex and SummaryIndex, then uses RouterQueryEngine to intelligently route 4 thematic synthesis queries to the best-fit engine.

✍️ Report Writer
gpt-4o · temperature=0.7 · as_structured_llm(ResearchReport)

Reads the full ChatMemoryBuffer history, then uses llm.as_structured_llm(ResearchReport) to generate a validated Pydantic report with title, executive summary, sections, conclusion, citations, quality score, and word count estimate.

Runner Modes

⚡ basic
python runner.py --topic "..." --mode basic

Runs the full workflow to completion and renders the final ResearchReport with rich Markdown panels. Uses asyncio.run(workflow.run(...)) — the simplest API. Best for production use.

📡 stream
python runner.py --topic "..." --mode stream --depth deep

Iterates async for event in handler.stream_events(). Each ProgressEvent emitted by ctx.write_event_to_stream() inside any step appears live as the workflow runs. Shows step-by-step progress.

🔍 debug
python runner.py --topic "..." --mode debug

Enables LlamaDebugHandler(print_trace_on_end=True). After the run, calls debug_handler.get_event_pairs(CBEventType.LLM) and renders a table of every LLM call with model, token counts, and prompt preview.

# Stream mode — live step-by-step output handler = workflow.run(topic=topic, depth=depth) async for event in handler.stream_events(): if isinstance(event, ProgressEvent): print(f" {event.step}: {event.message}") report = await handler # final result after stream ends

Pros & Cons

Strengths

  • Best-in-class RAG primitives (VectorStoreIndex, SubQuestionQueryEngine, RouterQueryEngine) built into the framework
  • 160+ pre-built integrations for LLMs, vector stores, document loaders, and data connectors
  • Event-driven Workflow avoids boilerplate routing tables — type annotations ARE the router
  • LlamaDebugHandler provides deep, token-level observability with no extra tooling
  • Fully async with native aquery(), achat(), astream() APIs
  • Structured output (as_structured_llm) works with any Pydantic model
  • Modular pip packages — install only what you need, no monolithic dependency
  • Works with any vector store backend (SimpleVector, FAISS, Pinecone, Chroma, Weaviate)

Limitations

  • Global Settings object can cause issues in multi-threaded or multi-tenant scenarios
  • No built-in human-in-the-loop (HITL) mechanism — requires custom interrupt logic
  • SubQuestionQueryEngine can be expensive (5+ LLM calls per complex query)
  • Workflow has no native checkpointing or session resumption (unlike LangGraph's MemorySaver)
  • No built-in hierarchical team management like CrewAI's manager agent pattern
  • Documentation can lag behind rapid package releases across 160+ integrations
  • Rapid API changes between minor versions can break integrations
  • Memory management is basic (ChatMemoryBuffer) vs. LangGraph's multi-store system
