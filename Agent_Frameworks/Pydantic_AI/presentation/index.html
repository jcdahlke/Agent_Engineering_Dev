What is Pydantic AI?

Created by Samuel Colvin and the Pydantic team — the same team behind Pydantic v2, which powers FastAPI, Django, SQLModel, and most of the Python AI ecosystem. Pydantic AI brings the same type-safety philosophy to building AI agents.

🔒

Type-Safe Agents

Agent[DepsType, OutputType] — fully generic. IDE autocomplete on every agent input and output, no runtime surprises.

💉

Dependency Injection

RunContext[DepsType] passed to every tool and system prompt. Inject HTTP clients, config, databases — no global state.

📦

Structured Outputs

Set output_type=YourModel and the agent guarantees a valid Pydantic model back — no string parsing.

🛠️

@agent.tool Decorator

Register tools on a specific agent instance. Tools receive RunContext for type-safe access to injected dependencies.

🔄

Output Validation + Retry

@agent.output_validator runs after each response. Raise ModelRetry to send feedback back and regenerate.

🌊

Streaming

async with agent.run_stream() as s: streams tokens incrementally. Full async/await throughout — production-grade.

System Architecture

Agents are called like regular async Python functions — no graph topology, no message bus. Context flows via deps=.

Typed Agent Declaration

# Fully typed — IDE knows DepsType and OutputType
researcher_agent: Agent[ResearchDependencies, ResearchFindings] = Agent(
    model=OpenAIModel("gpt-4o-mini"),
    output_type=ResearchFindings,
    deps_type=ResearchDependencies,
)

async def scrape_webpage(
    ctx: RunContext[ResearchDependencies],

    # Uses injected client — no global state
    resp = await ctx.deps.http_client.get(url)

Output Validator + ModelRetry

@analyzer_agent.output_validator
async def validate_analysis(
    ctx: RunContext[ResearchDependencies],
    result: AnalysisResult,
) -> AnalysisResult:
    if len(result.key_findings) < 3:
        raise ModelRetry("Need ≥3 findings")
    return result

Message History Chaining

# Researcher result carries its message history
res_result = await researcher_agent.run(
    prompt, deps=deps
)
# Analyzer receives full conversation context
ana_result = await analyzer_agent.run(
    prompt, deps=deps,
    message_history=res_result.new_messages(),
)

How It Works

1

Supervisor Plans

await supervisor_agent.run(topic, deps=deps) — returns a research plan string. Dynamic system prompt reads depth from ctx.deps.config.

2

Researcher Gathers

Executes DuckDuckGo, arXiv, Wikipedia, and scrape tools. Each tool injects ctx.deps.http_client. Appends URLs to ctx.deps.sources.

3

Analyzer Extracts

Receives content + message_history from step 2. Output validator enforces quality via ModelRetry. Returns typed AnalysisResult.

4

Writer Drafts

Chains .new_messages() from analyzer. Returns structured WrittenReport with title, sections, and citations. Loops if revised.

5

Critic Reviews

Scores 0-1. @output_validator enforces consistency. If score <0.7, sends revision instructions back to Writer (up to MAX_REVISIONS).

One shared ResearchDependencies instance flows through every step via deps=deps — single HTTP client, accumulated sources, session ID.

Unique Pydantic AI Patterns

Dependency Injection — the core pattern

# pipeline.py — one deps instance, shared across all 5 agents
async with httpx.AsyncClient(timeout=45.0) as http:
    deps = ResearchDependencies(
        http_client=http,      # shared — no per-request client creation
        config=config,          # typed settings, not env vars everywhere
    )
    # Every agent gets deps — fully typed, mockable in tests
    result = await researcher_agent.run(prompt, deps=deps)

Dynamic System Prompt

@researcher_agent.system_prompt
async def dynamic_prompt(
    ctx: RunContext[ResearchDependencies]
) -> str:
    # Reads config from injected deps at runtime
    depth = ctx.deps.config.research_depth
    return f"Depth: {depth}. Session: {ctx.deps.session_id}"

Streaming Output

# runner.py — stream mode demo
async with writer_agent.run_stream(
    prompt, deps=deps
) as stream:
    async for chunk in stream.stream_text(delta=True):
        print(chunk, end="", flush=True)

Pros & Strengths

  • 🔒
    End-to-end type safetyIDE autocomplete on ctx.deps.http_client, result.output.key_findings, every agent input and output. Runtime errors caught at declaration time.
  • 🐍
    Pythonic APIAgents are called like regular async def functions. No graph classes, no YAML, no DSLs — just Python you already know.
  • 🧪
    Excellent testabilityDependency injection means you swap ResearchDependencies(http_client=MockClient()) in tests. No monkeypatching, no global mocks.
  • 📦
    Guaranteed structured outputsoutput_type=PydanticModel means you always get a validated model back. No string parsing, no .get("key") on dict responses.
  • 🔄
    Built-in retry on validation failureModelRetry sends your error message back to the model, which regenerates. Elegant feedback loop with zero extra code.
  • 🌐
    Model-agnosticSwap OpenAIModel for AnthropicModel, GeminiModel, or any provider. Same agent code, different constructor.

Cons & Limitations

  • 🌱
    Newer, smaller ecosystemReleased in late 2024. Far fewer tutorials, community examples, and third-party integrations than LangChain or CrewAI.
  • 🗺️
    No built-in graph topologyYou wire the pipeline yourself in Python. Great for control, but LangGraph gives you visual state machines, persistence, and conditional branching out of the box.
  • 💾
    No built-in persistence or checkpointingLangGraph's MemorySaver / InMemoryStore and CrewAI's long-term SQLite memory have no equivalent. You bring your own storage.
  • ⚡
    Async-first complexityThe entire framework is async/await. For simple scripts or beginners, this is overhead. Calling sync code requires asyncio.to_thread().
  • 🔗
    Per-instance tool registration@agent.tool binds to a specific agent instance. Sharing the same tool across multiple agents requires duplicate wrapper functions.
  • 🤝
    No declarative agent handoffsOpenAI Agent SDK's handoff_to= and CrewAI's task routing handle orchestration for you. In Pydantic AI, you write the orchestration logic explicitly.

Use Cases & When to Choose

🏭

Production Agent APIs

Serving agents behind a FastAPI endpoint? Pydantic AI's native Pydantic types integrate seamlessly — request models become agent deps, response models become outputs.

🧪

Test-Driven Agent Development

Need 90%+ coverage? Dependency injection makes every agent trivially unit-testable. Inject mock clients, fake configs, and controlled tool responses.

🔗

Multi-Model Pipelines

Different agents on different providers with guaranteed typed handoffs? Agent[Deps, TypedOutput] ensures each step's output is the next step's typed input.

📋

Structured Data Extraction

Converting unstructured text to validated Pydantic models. The output validator + ModelRetry loop ensures you always get clean, schema-compliant data.

🏗️

Teams Already Using Pydantic

If your stack already uses FastAPI + Pydantic, Pydantic AI's models, validators, and settings patterns are immediately familiar — zero conceptual overhead.

🚫

When NOT to Choose It

Need persistence/HITL? Use LangGraph. Need zero-boilerplate multi-agent? Use CrewAI. Need native handoffs + tracing? Use OpenAI Agent SDK. Need RAG? Use LlamaIndex.
